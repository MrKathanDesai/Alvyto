# Alvyto Final Project Report — Part 2

## CHAPTER 4: SYSTEM REQUIREMENTS

### 4.1 User Characteristics

Alvyto’s primary users are **doctors and healthcare professionals** who conduct high-frequency outpatient consultations under strict time constraints. Their workflow requires rapid context switching between patient interaction, clinical reasoning, and documentation, often within appointment windows of 5–15 minutes. The system is therefore designed so that review and approval can be completed in under 60 seconds for most consultations: doctors can quickly verify highlighted findings, edit chips or bullets inline, regenerate narrative paragraphs if needed, and approve. Only basic computer literacy is required (mouse, keyboard, browser use), and no command-line or technical AI knowledge is expected.

The secondary user group is **clinic staff and medical assistants** who handle operational throughput: scheduling appointments, checking in patients, assigning rooms, monitoring queue progress, and ensuring doctor-room availability is aligned with daily load. These users primarily operate through the administrative dashboard and require fast, low-friction workflows that reduce clerical latency. Their interface priorities are data entry speed, clear status visibility, and operational integrity (e.g., linking check-in to queue creation and preserving appointment-room-doctor relationships).

The tertiary user group is **system administrators and technical operators** responsible for deployment, maintenance, and governance of the local clinic installation. Their responsibilities include managing Python environments for backend and room agent services, configuring and monitoring Ollama model runtime (including warm startup behavior), verifying service health across ports 3000/8080/8000, and handling backup/upgrade procedures. These users require stronger technical proficiency in infrastructure, service orchestration, and security controls than clinical users.

**Note:** **Patients are NOT direct users of Alvyto. They are indirect beneficiaries and data subjects of the medical records generated by the system.**

| User Group | Primary Responsibilities | Interface Used | Required Skill Level | Critical Outcome |
|---|---|---|---|---|
| Doctors / Healthcare Professionals | Conduct consultation, validate AI summary, approve final record | Room/doctor review UI | Basic computer literacy | Accurate, approved clinical record in under 60 seconds |
| Clinic Staff / Medical Assistants | Appointment scheduling, check-in, queue and room operations | Admin dashboard | Basic-to-intermediate office software skills | Smooth patient flow and reduced administrative delay |
| System Administrators | Deployment, model/runtime management, uptime/security monitoring | Server, service configs, admin APIs | Technical (Linux/Windows server ops, Python, networking) | Stable, secure, maintainable clinic deployment |

### 4.2 Functional Requirements (FR1–FR8)

**FR1 — Real-Time Audio Capture:** The system shall capture consultation audio continuously in real time from the browser and stream it to the room agent over WebSocket without manual chunk uploads. This is implemented through frontend recording controls and the `useWhisperLive` streaming hook, which sends audio chunks to the room agent service on port 8000 and supports stable long-running sessions up to 60 minutes. Operationally, this requirement is critical because consultation capture must not interrupt doctor-patient interaction or force additional post-visit recording tasks.

**FR2 — Speaker Diarization:** The system shall separate multi-speaker consultation audio into speaker clusters and allow clinical-role correction before summarization. Technically, pyannote.audio 3.x with ResNet34 embeddings performs clustering and silhouette-based adaptive speaker count, generating labels (`SPEAKER_00`, `SPEAKER_01`, etc.), followed by a manual `SpeakerConfirmation` step that uses `confirmSpeakerMapping()` for full transcript remapping. This matters clinically because role-aware extraction (doctor vs patient vs companion) is necessary to prevent misattributed findings and incorrect medical documentation.

**FR3 — Speech-to-Text Transcription:** The system shall transcribe clinical conversational audio with timestamped, alignment-aware output. This is satisfied using WhisperX `small.en` with INT8 compute type and word-level forced alignment, optimized for practical performance on clinic hardware while retaining adequate clarity for North American English speech patterns. This requirement is operationally essential because all downstream AI structuring, review, and record insertion depend on transcription fidelity.

**FR4 — AI-Based Medical Summarization:** The system shall generate structured medical summaries in strict JSON format from diarized transcript data. The implementation uses `phi4-mini` (3.8B, Q4_K_M) via Ollama, producing schema-constrained output that includes `clinicalSnapshot` (4–6 items), `doctorActions` (3–5 items), `issuesParagraph`, `actionsParagraph`, and related fields as raw JSON (without markdown fences). This requirement is clinically important because doctors need fast, structured draft documentation that can be reviewed quickly without free-text synthesis burden.

**FR5 — Hallucination Validation:** The system shall automatically validate whether extracted structured items are supported by transcript evidence before doctor approval. Validation is implemented using fuzzy matching rules: `SequenceMatcher` ratio > 0.72 and at least 60% token coverage for tokens longer than 3 characters, resulting in `isSupported` flags and visual warning states. This requirement is critical for patient safety and record integrity because unsupported statements must be prominently surfaced before becoming part of longitudinal medical records.

**FR6 — Doctor Review and Approval:** The system shall require explicit doctor review and approval before finalizing any visit summary. The review UI supports inline editing of chips and bullets, optional paragraph regeneration via `/expand`, and mandatory approval via `PATCH /api/visits/{id}/approve`; approval triggers final persistence and PHI clearing. Clinically, this requirement preserves physician authority and ensures the final record is medically accountable human-reviewed content.

**FR7 — Patient Record Management:** The system shall persist approved visit summaries and update patient longitudinal history consistently. During approval, structured conditions and medications are merged into `MedicalHistory` with case-insensitive deduplication, and visit records remain accessible as consultation history. This requirement matters because continuity of care depends on reliable, non-duplicative cumulative patient context across visits.

**FR8 — Administrative Module:** The system shall provide full clinic operations support for appointments, room assignment, queue tracking, doctor availability, auditability, and admin governance. This is implemented through CRUD modules (`patients`, `doctors`, `rooms`, `appointments`, `queue`), check-in flow (`/api/appointments/{id}/checkin`), visit oversight, `GET /api/audit-logs`, and super-admin-only admin user management endpoints. Operationally, this requirement is necessary to avoid fragmented tooling and to keep consultation capture integrated with day-to-day clinic throughput.

### 4.3 Non-Functional Requirements (NFR1–NFR6)

**NFR1 — Performance:** The system shall process consultations within clinically acceptable turnaround times on local hardware. Measured performance targets are 70–100 seconds end-to-end on CPU-only deployments, approximately 37 seconds on GPU-enabled setups, and `phi4-mini` warm inference around 7.6 seconds (vs ~30 seconds CPU warm). This is achieved through INT8 WhisperX inference, local Ollama execution, model pre-warming, and live preview streaming during capture so clinicians receive immediate in-session feedback.

**NFR2 — Accuracy:** The system shall maintain high transcription and structuring reliability suitable for physician-supervised clinical documentation. WhisperX `small.en` INT8 is used for practical conversational accuracy; `phi4-mini` benchmarked at 1.00 support rate with 100% JSON and schema compliance in evaluation set conditions. Residual errors are mitigated through mandatory review and inline correction workflows, ensuring final records remain clinician-verified.

**NFR3 — Security and Privacy:** The system shall enforce secure authentication and minimize PHI retention by design. Security controls include JWT HS256 authentication, bcrypt cost factor 12 for password hashes, SHA-256 token-hash storage in session records, server-side revocation checks, and controlled token lifetimes (8h admin, 24h room device). Privacy controls include clearing transcript artifacts (`transcript=""`, `dialogue=[]`) after approval and keeping `AuditLog` entries free of PHI narrative payloads.

**NFR4 — Reliability:** The system shall remain stable during full-length consultations and recover gracefully from transient disruptions. Room sessions are designed for up to 60-minute capture windows, frontend hooks manage WebSocket interruption handling, and room-agent background processing continues independently if a browser tab navigates away after upload completion. This ensures visit processing completion despite local UI disruptions.

**NFR5 — Usability:** The system shall be usable by clinicians and staff without technical training. Unsupported AI facts are highlighted in amber, transcript side-panel evidence is available for rapid fact-checking, and review actions (edit, regenerate, approve) are consolidated into a single focused workflow. These design choices minimize cognitive load and reduce time-to-approval.

**NFR6 — Maintainability and Scalability:** The system shall support modular maintenance, component replacement, and future scaling paths. Alvyto is split into three independent services (frontend, backend, room agent), allowing isolated updates and troubleshooting; Ollama model swapping can be done without major code rewrites if interface contracts are preserved. SQLite supports single-clinic operation today, with a clear migration path to PostgreSQL for multi-clinic or high-concurrency growth.

#### 4.4.3 Deployment Architecture

Alvyto follows a **hub-and-spoke local deployment model**. A single clinic server hosts all three services: frontend (port 3000), backend API (port 8080), and room agent AI service (port 8000). The architecture is intentionally local-first and can run without internet connectivity, reducing cloud dependency and external PHI transfer risk.

## CHAPTER 5: SYSTEM ANALYSIS

### 5.1 Study of Current System

**Manual Documentation:** In many clinics, consultation notes are still handwritten or typed manually after or during patient interaction. This forces doctors to divide attention between patient communication and record creation, reducing conversational quality and increasing omission risk. Documentation structure varies significantly by practitioner, causing inconsistency across records. Retrieval and longitudinal synthesis become difficult when narratives are non-standardized.

**Conventional EMR Systems (Epic, Cerner, OpenEMR):** Traditional EMR platforms digitize storage and workflow but still rely heavily on manual entry into templates and structured fields. They generally do not capture raw consultation dialogue as a first-class input source. Template-driven documentation can increase cognitive overhead when the clinician must navigate many fields under time pressure. As a result, completeness often depends on clerical effort rather than natural encounter flow.

**Voice Dictation Systems:** Dictation tools improve typing burden but typically operate post-consultation and produce flat transcripts. Without robust speaker separation, doctor and patient statements may be conflated in output. The clinician still must manually structure the text into assessment/action format and ensure record compliance. Net documentation burden decreases only partially.

**Cloud AI Transcription/Ambient Systems (Amazon Transcribe Medical, Nuance DAX, Abridge):** Cloud-native ambient systems can provide strong transcription accuracy and useful automation. However, they require internet dependence and external PHI transmission, which may conflict with strict local governance requirements. Ongoing subscription costs create recurring financial pressure for smaller clinics. Most do not provide tightly integrated in-clinic scheduling, queue, room, and approval governance in a single local platform.

### 5.2 Problems in Current System

1. **Inconsistent accuracy in conversational medical speech:** Clinical dialogue includes interruptions, colloquialisms, background noise, and mixed pacing. Generic transcription pipelines often degrade under these conditions, producing omissions and misrecognitions that require heavy correction.

2. **No speaker diarization (doctor and patient speech mixed):** Many systems output single-speaker-like transcripts, obscuring who said what. This directly affects correctness of symptom attribution, treatment decisions, and medico-legal clarity.

3. **No hallucination control before record entry:** AI summaries can include unsupported paraphrases or inferred details. Without a validation layer and visual cues, such content may pass into records unnoticed.

4. **Privacy risk from cloud PHI transmission:** External API-based processing transmits sensitive consultation data outside clinic-controlled infrastructure. This expands attack surface and introduces compliance and contractual risk.

5. **Fragmentation across tools:** Transcription, summarization, scheduling, and queueing are often handled in disconnected systems. This creates duplicate entry, synchronization errors, and operational inefficiency.

6. **Missing approval gate:** In some AI workflows, generated content can be accepted implicitly without structured physician sign-off. Lack of explicit approval undermines accountability and clinical governance.

### 5.3 Requirements of New System

1. The system must capture consultation audio in real time through the browser and process it locally. It must support uninterrupted sessions up to 60 minutes to match real outpatient variability.

2. The system must separate speakers using diarization and provide explicit doctor-side speaker confirmation. This ensures role-correct attribution before summary generation.

3. The system must generate schema-valid structured clinical JSON, not unstructured free text. Output must include clinically actionable sections with predictable fields for storage and review.

4. The system must implement hallucination support checks and flag unsupported statements. Unsupported items must be visually prominent and easy to audit against transcript evidence.

5. The system must enforce human-in-the-loop approval as a hard gate before persistence. No visit should be finalized without doctor confirmation.

6. The system must preserve patient longitudinal history by merging conditions and medications with deduplication. It should maintain visit-wise consultation records for continuity of care.

7. The system must integrate appointment, room, queue, and visit workflows in one platform. Check-in should connect directly to queue progression and room assignment.

8. The system must run on local clinic infrastructure with strong authentication, RBAC, and auditability. It should avoid mandatory cloud dependence and support future database scaling.

### 5.4 Process Model

Alvyto followed an **Incremental Agile-Waterfall Hybrid** process model. In this model, **Waterfall discipline** was applied at phase boundaries—requirements were formally locked, core architecture was fixed before broad implementation, and explicit test gates were used before advancing to the next stage. **Agile iteration** occurred inside each phase, where components were repeatedly refined through short feedback loops, especially around prompt behavior, UI review ergonomics, and integration reliability.

This model fit Alvyto because the product consists of three independently deployable services with distinct risk profiles. The highest technical uncertainty was the room-agent AI pipeline (capture → diarization → summarization → validation), so it was built and stabilized first to de-risk downstream dependencies. Once pipeline behavior was reliable, backend governance and frontend workflows were developed in parallel with clear interface contracts.

Seven validation gates were used: **(G1)** requirement baseline freeze, **(G2)** architecture and API contract sign-off, **(G3)** AI model benchmark and selection gate, **(G4)** room-agent end-to-end pipeline stability gate, **(G5)** backend security and RBAC gate, **(G6)** frontend review/approval usability gate, and **(G7)** full-system integration + acceptance gate. Each gate required objective pass criteria (schema compliance, latency ranges, transition guards, approval behavior, and security checks) before progression.

Compared with pure Waterfall, this approach avoided late discovery of integration failures that would have emerged only after full build completion. Compared with pure Agile, it prevented prolonged ambiguity in security boundaries, deployment architecture, and compliance-critical data lifecycle rules. The hybrid model balanced rigor for clinical governance with iterative optimization for model and UX behavior.

### 5.5 Feasibility Study

**Technical Feasibility:** The system is technically feasible using a fully open-source stack across frontend, backend, and local AI inference layers. Candidate model benchmarking established a measurable basis for selecting `phi4-mini` over alternatives using compliance and support metrics, not subjective preference. CPU-only viability was validated (70–100s end-to-end), confirming deployment feasibility in environments without dedicated GPUs.

**Operational Feasibility:** Browser-only room clients align with typical clinic IT constraints where endpoint maintenance must remain minimal. Clinical workflows map to existing habits: staff continue scheduling/check-in, doctors continue review/sign-off, and AI acts as acceleration rather than process replacement. The doctor review interface requires no specialized technical training.

**Economic Feasibility:** The architecture avoids per-request cloud API costs for core transcription and summarization by using local inference. Expected one-time server investment (~USD 800–1500 depending on GPU inclusion) is generally lower than recurring enterprise ambient-scribe subscription models over medium-term use. Open-source components reduce licensing burden and vendor lock-in risk.

**Schedule Feasibility:** A 13-week timeline is feasible through risk-first sequencing and modular service boundaries. High-risk AI pipeline concerns were addressed early, enabling parallel workstreams for backend and frontend once contracts stabilized. This reduced rework and kept critical milestones achievable.

### 5.6 Features of New System

#### 1. Real-Time Clinical Processing Pipeline
Alvyto captures live consultation audio directly from the room interface and streams it to the room agent without file-upload pauses. This allows the system to process speech continuously and provide immediate preview feedback during recording. The design reduces post-consultation documentation lag and keeps clinician focus in a single workflow. It also enables deterministic handoff from capture to diarization and summarization.

#### 2. Speaker-Aware Transcription and Structuring
The pipeline first separates speakers acoustically and then provides explicit role assignment confirmation by the doctor. This two-layer approach improves attribution reliability over single-pass transcription systems. Transcript segments are remapped globally using confirmed roles, and adjacent same-role turns are merged for readability. The result is cleaner clinical evidence mapping and safer extraction.

#### 3. Integrated Multi-Stage AI Pipeline
Alvyto combines VAD trimming, ASR, forced alignment, diarization, speaker-role mapping, summary generation, and validation in one orchestrated path. Each stage feeds structured outputs to the next with defined contracts and intermediate checkpoints. This minimizes fragile manual transformations between tools. Operationally, integrated orchestration improves reliability and debuggability.

#### 4. Structured Clinical JSON Generation
Instead of unstructured prose-only output, the system generates schema-constrained JSON with typed sections for facts, actions, findings, and prescription draft content. This structure supports deterministic rendering, validation, storage, and downstream export. It also enables clearer doctor review because each section has semantic purpose. Structured data readiness supports future analytics and interoperability.

#### 5. Zero-Inference Hallucination Guard
Every extracted fact is evaluated against transcript evidence before finalization using deterministic fuzzy support checks. Unsupported items are marked with `isSupported:false` and surfaced visually in amber for clinician attention. This guard does not attempt speculative correction and intentionally avoids adding inferred data. It reduces silent AI error propagation into records.

#### 6. Human-in-the-Loop Mandatory Approval
No AI draft can become a finalized medical record without explicit doctor approval. Clinicians can edit inline, regenerate narratives when needed, and then commit via approval endpoint. Approval updates visit and operational statuses consistently and logs accountable completion metadata. This preserves clinical authority and legal defensibility.

#### 7. PHI Minimization by Design
Raw transcript artifacts are treated as temporary processing context rather than permanent record content. After approval, transcript and dialogue fields are cleared while structured approved outcomes are retained. Audit logs capture operational events but exclude PHI narrative text. This approach narrows long-term PHI exposure surface.

#### 8. Unified Clinical and Administrative Platform
Alvyto combines patient, doctor, room, appointment, queue, visit, and approval flows in one coherent system. Staff and clinicians no longer need disconnected tools for operational and documentation tasks. Data continuity between scheduling and consultation completion is maintained by design. This reduces duplication and process errors.

#### 9. CPU-Optimised Local Deployment
The pipeline is validated for CPU-only operation, enabling deployment in clinics without dedicated accelerators. GPU support improves throughput but is optional, preserving accessibility across infrastructure tiers. Local inference also removes mandatory internet dependence for core operation. This broadens practical adoption potential.

#### 10. Modular Three-Service Architecture
Frontend, backend, and room agent are independently deployable services with clear boundaries. Failures, updates, and optimizations can be isolated to one service without full-system rewrites. Model swaps through Ollama remain possible without changing core business workflow modules. The architecture supports maintainability and controlled evolution.

## CHAPTER 6: DETAILED DESCRIPTION

### 6.1 Doctor / User Module

#### 6.1.1 Room Session Initialization
Room sessions begin through authenticated room-device access using room ID + PIN submitted to `POST /auth/room-login`. The backend issues a JWT token scoped for room-device operations with a 24-hour expiry window, separate from admin-token policy. This scoping ensures room clients can perform consultation tasks without inheriting administrative privileges. The workflow keeps authentication simple for room usage while preserving RBAC separation.

#### 6.1.2 Live Audio Capture and Preview
A single primary control starts consultation recording in the room UI. Audio chunks are streamed over WebSocket to the room agent on port 8000 via the `useWhisperLive` hook, which handles session state, streaming lifecycle, and event updates. During capture, incremental transcription preview is displayed so clinicians can monitor intake quality in real time. This immediate feedback reduces uncertainty and allows early correction if capture quality degrades.

#### 6.1.3 Speaker Confirmation
After transcription and diarization complete, a `SpeakerConfirmation` overlay presents sample text snippets per detected cluster (`SPEAKER_XX`). Initial order-based role hints come from engine logic (first speaker → Doctor, second → Patient, third → Companion), but the doctor can correct mappings before summary generation. `confirmSpeakerMapping()` applies remapping across all segments, and `mergeConsecutiveSpeakers()` collapses adjacent same-role turns for cleaner narrative flow. This step closes the gap between acoustic clustering and clinically correct role attribution.

#### 6.1.4 Summary Review Interface
The review view presents `clinicalSnapshot` as editable chips, `doctorActions` as editable bullets, and `issuesParagraph`/`actionsParagraph` as narrative text blocks. Items with `isSupported:false` are amber-highlighted to direct physician attention toward low-confidence or unsupported content. A collapsible transcript side panel provides evidence context, and hovering a chip can highlight corresponding transcript turns for rapid verification. The interface is optimized for high-speed, low-friction medical validation.

#### 6.1.5 Inline Editing and Paragraph Regeneration
Doctors can directly edit chips, bullets, and draft narrative text inline without navigating away or opening a separate edit mode. Small corrections are immediate and do not require model re-inference, reducing review latency. When broader narrative rewrites are needed after factual edits, the `/expand` endpoint can regenerate paragraph-level prose from updated structured points. This hybrid edit strategy balances speed and consistency.

#### 6.1.6 Approval and PHI Minimization
On approval, the client calls `PATCH /api/visits/{id}/approve` to finalize documentation. The backend persists structured summary JSON, extracts/patches chief complaint if missing, merges conditions and medications into `MedicalHistory` with case-insensitive deduplication, stamps `approved_at`/`approved_by`, and marks visit and linked operational entities complete. It then clears raw PHI transcript payload (`transcript=""`, `dialogue=[]`), marks appointment complete, updates queue entry to done, and releases room assignment to idle. This transactional closeout integrates clinical completion with privacy minimization.

### 6.2 System / Backend Module

#### 6.2.1 Data Architecture
The backend persists clinical and operational state through core SQLAlchemy entities: `AdminUser`, `Doctor`, `DoctorAvailability`, `Room`, `Patient`, `MedicalHistory`, `Appointment`, `WaitingQueue`, `Visit`, and `AuditLog`, with `Session` supporting authentication integrity. Key relationships include Patient 1:1 MedicalHistory, Patient 1:N Visit, Doctor N:N Room through availability mapping, and Appointment linking Patient + Doctor + Room (+ Visit association during lifecycle). WaitingQueue references Patient, Appointment, Room, and Doctor for real-time flow tracking. AuditLog is append-only for governance events.

#### 6.2.2 Authentication and Session Management
Authentication uses JWT HS256 access tokens generated by token creation utilities (e.g., `create_access_token`) and validated server-side per request. Session persistence stores only SHA-256 token hashes (not raw token values), alongside expiry and revocation state, and passwords use bcrypt cost 12. RBAC defines three scoped roles with permission checks at API routes, and logout triggers revocation (`revoke_session()`), while request guards (`is_session_valid()`) enforce expiry/revocation constraints continuously. This ensures access control is backend-enforced rather than UI-trust-based.

#### 6.2.3 Visit Lifecycle
Visit state changes are strictly controlled by backend transition validation. Allowed transitions are `pending → in_progress/cancelled` and `in_progress → completed/cancelled`; `completed` and `cancelled` are terminal. The `ALLOWED_TRANSITIONS` map is checked in status routes, and invalid transitions are rejected with explicit errors. This prevents workflow corruption and ensures auditable, deterministic visit state progression.

#### 6.2.4 Room Agent (Port 8000)
The room agent runs a background in-memory job queue with a single worker thread to serialize processing and simplify local reliability. Processing sequence is: audio receive → VAD trim → WhisperX transcription → word-level alignment → pyannote diarization → silhouette speaker count resolution → transcript merge/remap → phi4-mini summarization → fuzzy support validation → result persistence and frontend notification. Models are pre-warmed at startup to reduce first-request latency penalties. This architecture provides predictable local inference behavior with manageable operational complexity.

#### 6.2.5 Prescription System
Prescription drafting is embedded within `VisitSummary.prescriptionDraft`, including diagnoses, medications (name/dosage/frequency/duration/route/instructions), investigations (name/details/timing), advice, warnings, reportSummary, and follow-up plan (timeline/notes). Backend HTML rendering is generated via `_build_prescription_html()` for printable/downloadable output, while frontend uses `pdf-lib` and helpers such as `normalizeDraft()` and `triggerPrescriptionDownload()` for resilient PDF export. This allows clinically usable prescription output even when some legacy payload variations require normalization. The approach maintains a structured source of truth while supporting document-grade rendering.

### 6.3 Administrator Module

#### 6.3.1 Patient Management
Administrators can create, update, retrieve, and manage patient records through `/api/patients` CRUD endpoints. Demographic and baseline clinical references are centralized for use in appointment and visit flows. Patient records are linked to medical history and consultations to preserve longitudinal continuity. This module is the entry point for identity-consistent care documentation.

#### 6.3.2 Doctor Management and Availability
Doctor profiles are maintained through `/api/doctors`, while schedule and room compatibility are governed through availability mappings. Availability data drives assignment validity for appointments and room operations. This ensures the system respects practical staffing constraints during planning and execution. Accurate doctor availability directly reduces scheduling conflicts.

#### 6.3.3 Room Management
Rooms are managed through `/api/rooms` with status and assignment semantics. Operational states (including idle/occupied patterns) are synchronized with visit and queue transitions. Room metadata enables staff to allocate capacity and avoid collisions in active clinic hours. This module provides physical resource visibility in real time.

#### 6.3.4 Appointment Scheduling
Appointments are handled via `/api/appointments` CRUD operations with explicit patient-doctor-room linkage. The check-in endpoint (`PATCH /api/appointments/{id}/checkin`) bridges scheduled intent into live operational flow by activating queue participation. This conversion from planned to active state is essential for accurate day-of-clinic throughput. It also anchors downstream visit creation and completion accountability.

#### 6.3.5 Waiting Queue Operations
Queue management (`/api/queue`) tracks consultation progression through statuses such as `waiting → called → in_room → done`, with priority levels (1–4) available for urgency handling. Queue records preserve links to patient, appointment, room, and doctor for traceable handoff. Staff can monitor and adjust real-time flow without breaking entity relationships. This module enables predictable patient movement during peak load.

#### 6.3.6 Visit Oversight
Administrative users can list visits (`GET /api/visits`, admin-only), inspect status, and review summary-level completion signals. This provides supervisory visibility over in-progress and completed encounters. Soft delete semantics (`is_deleted=True`) preserve auditability while supporting operational correction workflows. Oversight supports governance and reporting functions.

#### 6.3.7 Audit Log Viewer
The audit module (`GET /api/audit-logs`) provides paginated, filterable event trails for sensitive operations and lifecycle transitions. Logs are designed for governance traceability, not narrative clinical storage. PHI-rich transcript narrative is excluded from audit payload content. This separation supports compliance and forensic review needs.

#### 6.3.8 Admin User Governance
Super-admin users can create and manage admin accounts via `/api/admin-users` endpoints (GET/POST/PATCH). Role assignment controls privilege boundaries across administrative actions, ensuring least-privilege operations. This governance layer centralizes identity lifecycle for non-clinical system operators. It also supports controlled delegation in larger clinic teams.

### 6.4 Inter-Module Workflow Integration
A typical daily clinic flow proceeds as follows: an appointment is created and associated with patient, doctor, and room; at arrival, the patient is checked in and queue entry is activated; when called, the patient is moved to room context where consultation audio is captured and processed by the room agent; the doctor reviews structured outputs, verifies flagged evidence, and approves; approval finalizes visit status, updates medical history, marks queue and appointment complete, and releases room resources back to idle. This integrated path ensures operational continuity from front desk scheduling to clinically approved record persistence.

### 6.5 Error Handling and Reliability Controls
Backend routes enforce strict input validation and lifecycle guards. Payload schemas are validated, route-level permissions are checked, and status transitions are constrained by server-side rules to prevent invalid state mutations. Approval logic executes multi-entity updates in controlled order to avoid partial completion drift.

Frontend hooks are designed for recovery-oriented behavior under real clinic conditions. Streaming/session hooks tolerate transient disconnections, preserve user-visible progress states, and attempt safe continuity where feasible. UI state transitions expose clear actionability rather than silent failure, reducing operator confusion.

The system also supports legacy data tolerance through response shaping and sanitization layers. Older or partially variant payload structures are normalized before rendering, especially in summary and prescription paths. This prevents brittle failures during migration or historical data reads and preserves operational continuity.

### 6.6 Security Controls and Governance
Authentication and RBAC are enforced at API level rather than relying on UI visibility toggles. Every sensitive route validates token, role, and session state before permitting access, including super-admin-only governance endpoints. This ensures privilege boundaries remain intact even if client code is modified.

PHI minimization is implemented through lifecycle design: raw transcripts support processing and review but are cleared after approval, while structured approved records are retained. Audit logs intentionally avoid storing PHI narrative content, reducing risk exposure in governance trails. This approach aligns usability with privacy-by-design principles.

Session integrity is maintained through token hashing, expiry checks, and revocation controls on each request. Logout operations revoke server-side session validity immediately, and compromised tokens can be invalidated centrally. Together, these controls reduce replay risk and strengthen access governance over time.

## CHAPTER 7: TESTING

### 7.1 Testing Objectives

Testing was designed to validate four quality dimensions: **functional correctness** (whether each endpoint and workflow behaves according to requirements), **integration consistency** (whether multi-module transitions remain synchronized end-to-end), **RBAC reliability** (whether role-based restrictions are enforced server-side), and **AI model quality** (whether transcription/summarization outputs meet structure, support, and latency constraints). The objective was to ensure Alvyto is safe, stable, and operationally fit for real clinic deployment.

### 7.2 Testing Methodology

A layered strategy was used: black-box testing for externally observable behavior, white-box testing for internal logic paths and guards, integration testing for cross-module lifecycle continuity, and build-quality checks for regression control across frontend/backend services. In parallel, a model benchmark framework evaluated candidate LLMs on structured output reliability, schema compliance, hallucination support rate, inference speed, and CPU feasibility. This combined method balanced user-facing correctness with internal implementation integrity.

### 7.3 Test Environment

Testing was conducted in a three-service local environment mirroring production topology: Next.js frontend on port 3000, FastAPI backend on port 8080 with SQLite, and FastAPI room agent on port 8000 with WhisperX, pyannote.audio, and Ollama. Workflow simulation used synthetic and recorded consultation audio with varying speaker patterns to test diarization and approval behavior under realistic turn-taking conditions. API validation included role-specific tokens (super_admin/admin/room_device) to verify permission boundaries and session lifecycle behavior.

### 7.4 Black Box Testing

Black-box tests focused on behavior visible at endpoint or UI boundaries, independent of implementation internals. The goal was to confirm that each user-facing operation returns correct outputs, rejects invalid requests appropriately, and preserves expected workflow semantics.

Coverage emphasized high-risk clinic workflows: authentication, scheduling/check-in/queue, visit lifecycle transitions, doctor approval, transcript minimization, and governance endpoints. Result classification used pass/fail based on expected status codes, state changes, and visible data integrity outcomes.

### 7.6 Integration Testing

Integration testing validated the complete clinic execution path across all modules: appointment creation, patient check-in, queue activation, visit initiation, live processing updates, doctor approval, and final completion updates. The central assertion was that one action’s output becomes the next action’s valid input without orphan states or manual correction. End-state checks verified synchronized updates across Visit, Appointment, WaitingQueue, and Room entities.

A specific integration concern around **schedule-to-queue linkage** was tested and corrected: check-in must deterministically create/activate queue context bound to the same patient-doctor-room appointment tuple. Validation confirmed that once this linkage logic was fixed, downstream visit creation and room-release completion executed consistently without cross-entity mismatch.

#### 7.7.4 Key Findings

**qwen2.5:3b:** This model delivered fast responses (average 7.1s) and perfect JSON/schema compliance in benchmark cases. However, support rate was 0.93 due to paraphrase-risk deviations in Diabetes (0.89) and Sore Throat (0.90) cases where wording drifted from transcript-grounded phrasing. In a safety-sensitive record pipeline, this non-verbatim extraction risk was considered unacceptable despite speed advantages. The model was therefore rejected.

**qwen3:4b:** qwen3:4b consistently failed to emit valid structured JSON across all benchmark cases, yielding 0% JSON and schema compliance with support rate 0.00. Observed behavior indicated hybrid-thinking token consumption that exhausted generation budget before compliant JSON output was produced. Because deterministic schema output is non-negotiable for Alvyto’s pipeline, this model was disqualified at benchmark stage.

**phi4-mini:** phi4-mini achieved 1.00 support rate across all cases and 100% JSON/schema compliance, satisfying core correctness and structure constraints. Although initial cold-start latency reached 25.4s, warm inference stabilized at 7.6s with startup pre-warming, resulting in practical average performance of 12.1s over test conditions. Its balance of reliability, support quality, and manageable speed made it the selected model for production.

**llama3.1:8b:** llama3.1:8b also achieved 1.00 support and full structural compliance, indicating strong quality output. However, average inference time of 15.3s under CPU-constrained clinic deployment targets exceeded acceptable throughput expectations compared with phi4-mini warm behavior. Given equal quality but weaker operational fit, it was rejected for current deployment profile.
