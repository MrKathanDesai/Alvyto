ABSTRACT
Alvyto is a local clinical documentation system that converts doctor patient conversations
into structured medical records. During a consultation, the system captures room audio,
turns speech into text, separates doctor and patient speech, and prepares a draft summary for
review. The doctor checks the draft, edits it if needed, and approves it before it is saved in the
patient record.
The platform has three connected parts: a web interface for doctors and staff, a backend
service for records and access control, and a room service for audio processing. It also
includes appointment scheduling, queue handling, and room management so that daily clinic
work stays in one place.
The system is designed for local use inside the clinic network. Sensitive consultation data is
not kept longer than needed, and all critical actions are recorded in an audit log. Internal
testing shows that the full consultation to summary flow works reliably on standard clinic
hardware, with faster performance when graphics acceleration is available.
This project shows that practical, local-first clinical documentation is possible with clear
doctor control over final records.

INTRODUCTION

1.1 PROJECT SUMMARY
Alvyto is a local clinical documentation platform built for day to day consultation work
inside a clinic. During a doctor patient consultation, the system captures room audio,
converts speech to text, separates doctor and patient speech, and prepares a structured visit
draft. The doctor reviews and edits the draft, then approves it before it is saved to the patient
record.
The project is built as three connected services:
1. Frontend in Next.js for room and admin interfaces
2. Backend in FastAPI for records, authentication, and workflow APIs
3. Room agent for audio processing and summary generation
The admin side includes patient records, doctors, rooms, scheduling, queue operations, visits,
audit logs, and admin user management.

1.2 PROJECT PURPOSE
The project solves a practical clinic problem: doctors spend too much time typing notes and
handling repeated record entry steps after every consultation.
Alvyto reduces this effort by creating a draft from spoken consultation data and keeping the
doctor in control of final approval. This improves speed, supports better structure in records,
and keeps consultation and administration in one workflow instead of spreading work across
disconnected tools.

1.5 TECHNOLOGY AND LITERATURE OVERVIEW
●​ WhisperX:WhisperX

extends

Whisper

with

forced

●​ Pyannote.audio:pyannote.audio performs speaker diarization using deep speaker
embeddings and clustering. Alvyto uses ResNet34 embeddings with silhouette-based
adaptive speaker count so the pipeline does not hardcode two speakers. This avoids
mislabeling when a third voice (assistant or family member) enters a consultation.
Diarized segments are merged with aligned ASR words for downstream structured
extraction.
●​ phi4-mini:phi4-mini (3.8B, Q4_K_M) was selected after comparative benchmarking
against qwen2.5:3b, qwen3:4b, and llama3.1:8b. It produced 1.00 support rate and
100% schema compliance across benchmark scenarios. qwen3:4b was disqualified
due to token-budget consumption by hybrid reasoning traces before valid JSON
output. phi4-mini offers quality-speed balance suitable for clinic latency constraints.
●​ FastAPI:FastAPI

powers

backend

APIs,

authentication,

and

room-agent

orchestration interfaces. The backend implements JWT HS256, bcrypt-12 password
hashing, RBAC roles, and append-only audit logging. SQLAlchemy models maintain
structured patient, visit, queue, and administration data. FastAPI was chosen for
high-performance async endpoints and clean schema-driven development.
●​ Next.js with TypeScript:The frontend provides the doctor workflow (recording,
transcript review, summary editing, approval) and administrative dashboard.
Role-aware routing restricts room devices to room-scoped operations. UI behavior
includes amber highlighting for unsupported summary items and mandatory approval
gating before commit. Browser-only clients keep room hardware lightweight and easy
to maintain.

1.6 SYNOPSIS
In each consultation, room audio is streamed from browser to room agent over WebSocket,
transcribed and aligned by WhisperX, diarized by pyannote.audio, summarized by phi4-mini,
and validated against transcript evidence under strict support thresholds; the doctor then edits
or approves the output, after which the backend commits structured records, updates patient
history, logs the event in immutable audit trails, and erases raw transcript content to enforce
PHI minimization.

LITERATURE SURVEY

2.1 SURVEY DESIGN AND SELECTION CRITERIA
The survey was structured to evaluate technologies and systems relevant to consultation
documentation under clinic constraints. Selection criteria included:
- local deployability and operational control
- support for speaker-aware transcription workflows
- structured output suitability for record systems
- integration complexity with backend clinical workflows
- reliability of output format under repeated runs
- suitability for hardware commonly available in clinics

2.2 TECHNICAL FOUNDATIONS REVIEWED
2.2.1 Speech Transcription
Whisper established strong baseline robustness in noisy, multi-accent speech settings [1].
WhisperX extends this by adding alignment and practical integration patterns needed for
timestamp-sensitive transcript processing [2].
2.2.2 Speaker Separation
pyannote.audio provides a production-usable toolkit for speaker diarization and is commonly
used to separate speaker turns in conversational audio [3]. This directly supports consultation
formatting where attribution between doctor and patient is necessary.
2.2.3 Structured Draft Generation
For project-specific model selection, the report benchmarked four candidates:
- qwen2.5:3b
- qwen3:4b
- phi4-mini
- llama3.1:8b
The selected model in your implementation is phi4-mini, based on internal benchmark
outcomes documented in Chapter 7.

2.4 GAP IDENTIFIED FROM SURVEY
The survey indicates that many available tools solve only part of the workflow:
- transcription-only systems do not complete record-approval flow
- record systems do not generally generate consultation drafts from live room audio
- administrative tools (scheduling, queue, room operations) are often separated from
documentation processing
Alvyto addresses this gap through one integrated local platform that connects consultation
processing and clinic operations while preserving doctor approval control.

2.5 JUSTIFICATION OF TECHNOLOGY CHOICE
The selected architecture and toolchain are justified by project constraints:
- local execution requirement
- need for timestamp-aware transcript handling
- requirement for speaker-separated consultation text
- requirement for structured draft output integrated with backend schemas
- requirement for practical deployment and maintainability in clinic infrastructure

PROJECT MANAGEMENT

3.1 PLANNING STRATEGY
3.1.1​ Software Scope
The development approach for Alvyto followed an iterative, module-first plan. The project
was divided into independent services so that each critical part could be developed and
validated before full integration.
The three service units were:
1. frontend service for room and admin interfaces
2. backend service for records, access control, and operational APIs
3. room processing service for consultation audio pipeline
This plan reduced integration risk, especially because consultation processing had higher
uncertainty than standard admin modules.

3.2 DEVELOPMENT PROCESS MODEL
The appropriate process model for this project is Iterative Incremental Development with
Validation Gates.
This model fits Alvyto better than a strict linear model because:
- transcription quality, speaker separation quality, and draft quality need repeated
tuning
- frontend workflow and backend schema must evolve together
- clinic usability issues appear during practical trial runs, not only in initial design
3.2.1 Validation Gates used in practice
- Gate 1: room audio capture and stable transcript stream
- Gate 2: speaker-separated transcript consistency
- Gate 3: structured draft generation with schema-valid output
- Gate 4: doctor review and approval workflow completion
- Gate 5: admin operations (scheduling, queue, rooms, visits) integrated with backend
This gate-based progression reflects how the project was actually built and tested.

3.5 RISK MANAGEMENT
3.5.1 Method Used for Risk Identification
Risk assessment was performed against the implemented system, not against a generic
template.
The review considered:
●​ the three-service architecture (frontend, backend, room service)
●​ consultation flow (capture, transcript, structured draft, doctor approval)

Each risk was evaluated using:
1. trigger condition in this project
2. impact on correctness, usability, or safety
3. mitigation currently present in code and workflow
4. remaining exposure and next control

3.5.3 In-Depth Analysis by Risk Group
A. Technical Pipeline Risks (R1 to R5)
●​ R1 Alignment drift
In Alvyto, correctness depends on consistent transfer between transcript
segments, speaker labels, and structured draft fields. Any mismatch can
distort clinical meaning. Current control is a strict review gate and
normalized intermediate handling, which reduces unsafe persistence risk
but does not eliminate draft quality issues.
●​ R2 Streaming segment loss
Room capture is sensitive to network instability and session interruptions.
If early or mid-consultation segments are lost, draft completeness
decreases. Because final save is doctor-controlled, harmful persistence is
reduced, but review workload increases.
●​ R3 Context fragmentation

Long encounters can spread one clinical thought across multiple transcript
windows. This can reduce continuity in generated draft sections. Current
implementation allows manual correction and summary refinement, but
this remains a quality risk in long, complex consultations.
●​ R4 Startup latency
Initial request latency is practical in observed workflow but still noticeable
in cold scenarios. This is a usability risk rather than a correctness risk.
Warm behavior after first run is better.
●​ R5 Write pressure
Unlike generic designs that write every raw stream event, Alvyto limits
final persistence points. This reduces write load. Still, high operational
activity in queue and scheduling can create pressure under heavy usage.
B. Integration and State Risks (R6 to R8)
●​ R6 Service coupling impact
The architecture is decoupled enough that admin functions can continue
even if room draft flow is degraded. This is a strong design choice for
operational resilience.
●​ R7 Session state consistency
Encounter state must remain coherent from capture through approval.
Previous hardening work on state handling and normalization reduced this
risk, but interruption recovery remains important.
●​ R8 Sync lag and ordering
Real-time interfaces naturally face asynchronous update timing. Current
hook-level corrections reduce visible errors. Residual impact is moderate
and mainly affects user trust in live view timing.

●​ R9 Clinical misinterpretation
This is one of the highest-impact risks. Even when wording appears correct,
clinical framing can be wrong. Alvyto’s key safeguard is that no draft enters

end-to-end workflow continuity from consultation capture to doctor-approved record and
operational queue flow within a clinic-first system.

●​ variable integration complexity between note-generation products and existing

6.1 CONSULTATION AND DOCTOR MODULE
This module handles the complete consultation-side flow from room session start to final
approved visit record. It is designed to reduce manual typing work while keeping the doctor
in full control of final clinical content
6.1.1 Session Initialization in Room Context
Each consultation begins in a room-scoped authenticated context. This ensures the
system always knows:
●​ which room is active
●​ which patient is being consulted
●​ which visit record is being updated
The room-scoped session prevents cross-room data mixing and keeps consultation
actions tied to the correct operational context.
6.1.2 Live Audio Capture and Transcript Build
During consultation, room audio is streamed and converted into text segments. The
transcript is maintained with speaker separation so doctor and patient lines are not
merged into a single unstructured block. This improves review quality and reduces
confusion during post-consultation verification.
In practical terms, the doctor is not required to type full notes while interacting with
the patient. The system collects the spoken interaction and prepares it for structured
review.
6.1.3 Structured Draft Preparation
After sufficient transcript content is available, the system prepares a structured draft.
The draft format is designed for clinical readability and includes key sections used in
visit documentation. Instead of a single paragraph, output is grouped so the doctor
can quickly verify core points and treatment actions.
This stage is a draft support stage only. It is not the final record.

LIMITATIONS AND FUTURE ENHANCEMENT

9.1 LIMITATIONS
The current version of Alvyto is functional for the defined project scope, but several
limitations remain. These limitations should be stated clearly so the report stays technically
honest and academically strong.
9.1.1 Limited Scale Validation
The system has been validated on project-level workloads, but large concurrent
multi-room stress behavior is not yet fully benchmarked. In practical terms,
performance

under heavy simultaneous sessions,

especially on

CPU-first

deployments, still needs deeper capacity testing.
9.1.2 Internal Evaluation Scope
Model and workflow evaluations are based on internal test scenarios. While useful for
implementation decisions, this is not equivalent to broad external clinical validation.
More diverse datasets and independent evaluation settings are needed for stronger
generalization claims.
9.1.3 Non-Verbal Clinical Context Not Fully Captured
The current consultation pipeline processes spoken content but does not directly
capture non-verbal cues such as gesture, tone shifts, or silence patterns. In certain
consultations, these cues may influence interpretation and should be documented
manually by the clinician when needed.
9.1.4 Interoperability Boundary
The system currently operates as a clinic-local platform and does not yet provide full
production interoperability with external hospital ecosystems through standardized
exchange layers such as complete HL7 FHIR integration [22],[ 23].
9.1.5 Security and Compliance Maturity Level
Role-based access control and auditability are implemented, but full enterprise
hardening and formal compliance packaging are outside this project phase. Examples
include end-to-end deployment hardening, formal security verification cycles, and
complete regulatory documentation [24], [25].

10.1 CONCLUSION
This project set out to build a practical, clinic-oriented documentation and operations system
that supports the real consultation workflow instead of treating transcription as an isolated
feature. The implemented system combines consultation capture, speaker-separated transcript
processing, structured draft preparation, doctor review and approval, and administrative
operations in one connected platform.
From an implementation perspective, the project achieves the following within its defined
scope:
1)​ a complete consultation-to-record workflow with mandatory doctor approval before
final persistence
2)​ integrated appointment, check-in, queue, room, and visit lifecycle management
3)​ role-based operational control and auditable action flow
4)​ modular architecture across frontend, backend, and room-processing services
5)​ internal benchmark-driven selection of the draft-generation model used in
deployment
A major outcome of the project is workflow continuity. Scheduling, queue state, room
operations, and visit completion are now linked instead of handled as disconnected tasks.
This reduces repeated manual coordination and improves operational clarity for clinic staff.
Another important outcome is quality control through clinician oversight. The system does
not treat generated drafts as final records by default. The doctor remains the final
decision-maker through a required review and approval step. This preserves clinical
responsibility while still reducing documentation effort.
Testing results within project scope indicate stable behavior across core modules, including
access control, status transitions, integration paths, and frontend build quality checks. Defects
identified during development were addressed through targeted backend validation
improvements and interface-level resilience updates.
At the same time, the project is presented with clear boundaries. High-concurrency scale
behavior, broader external validation, and full enterprise hardening remain next-stage work.
These are not hidden, and they define a realistic roadmap for future maturity.
In summary, Alvyto demonstrates that a local, modular, doctor-controlled consultation
documentation system can be implemented with integrated clinic operations and practical

Implemented or recommended practices include:
●​ input validation and schema-safe parsing
●​ strict status transition guards
●​ explicit error codes for invalid operations
●​ idempotent-safe handling where applicable
●​ auditability for privileged actions
11.3.6 Documentation Standard
API behavior should be maintained through versioned endpoint documentation and
request/response examples.
Where OpenAPI descriptions are generated, they should be aligned with actual route
behavior and updated with each release [28].
